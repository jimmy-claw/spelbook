# lez-registry

**On-chain Program Registry for the LEZ ecosystem.**

`lez-registry` is a standalone LEZ program that provides an on-chain directory for LEZ programs. Developers register their deployed program IDs along with human-readable metadata and a Codex CID pointing to the program's IDL JSON stored on Logos Storage.

Built with [lez-framework](https://github.com/jimmy-claw/lez-framework) — the Anchor-inspired developer framework for LEZ programs.

---

## What it does

- **Register** a new program: anyone can register their program. The signer becomes the author and only they may later update the entry.
- **Update** an existing entry: only the original author (signer) is allowed to update metadata.

On-chain state stored in PDAs:
- `RegistryState` — singleton tracking global registry statistics (program count, authority)
- `ProgramEntry` — per-program metadata (name, version, author, IDL CID, description, tags, registration timestamp)

## Crate structure

```
lez-registry/
├── registry_core/       # Shared types: Instructions, ProgramEntry, RegistryState, PDA helpers
├── registry_program/    # On-chain handlers: register.rs, update.rs (with unit tests)
├── methods/             # risc0 build crate
│   └── guest/           # zkVM guest binary — #[nssa_program] / #[instruction] macros
├── cli/                 # CLI wrapper (3-liner using lez-framework-cli)
│   └── src/bin/
│       ├── registry.rs      # lez_framework_cli::run().await
│       └── generate_idl.rs  # lez_framework::generate_idl!(...)
├── e2e_tests/           # End-to-end tests (placeholder, runs against local sequencer)
├── registry-idl.json    # IDL schema (generated by `make idl`, committed for reference)
└── Makefile             # build, check, idl, cli, test, deploy targets
```

## Relationship to lez-multisig

`lez-registry` is **standalone** — no dependency on `lez-multisig`. The multisig is a _consumer_ of the registry (it can register itself), not part of it.

## Quick start

```bash
# Check that host crates compile (no risc0 needed)
make check

# Build the zkVM guest binary (requires risc0 toolchain)
make build

# Generate the IDL from #[nssa_program] annotations
make idl

# Run unit tests
make test

# See all available CLI commands (auto-generated from IDL)
make cli ARGS="--help"
```

## CLI usage

The CLI is IDL-driven — commands are auto-generated from `#[nssa_program]` / `#[instruction]` annotations. After running `make idl`:

```bash
# Register a program
make cli ARGS="-p <binary> register \
  --name lez-multisig \
  --version 0.1.0 \
  --idl-cid <codex-cid> \
  --description 'Multi-signature wallet' \
  --tags governance,multisig \
  --program-id 0,0,0,0,0,0,0,1"

# Update an existing entry
make cli ARGS="-p <binary> update \
  --program-id 0,0,0,0,0,0,0,1 \
  --version 0.2.0"

# Inspect program binary (show ProgramId)
make cli ARGS="inspect <binary>"

# Dry run without submitting
make cli ARGS="--dry-run -p <binary> register ..."
```

Or use the `registry` binary directly:
```bash
registry --idl registry-idl.json -p registry.bin register --help
```

## IDL

`registry-idl.json` describes the program interface. Generate fresh with:

```bash
make idl
```

## Architecture

- **`#[nssa_program]`** macro in `methods/guest/src/bin/registry.rs` defines the on-chain program
- **`#[instruction]`** handlers delegate to `registry_program::register::handle` / `::update::handle`
- **`generate_idl!`** reads those annotations at compile time → emits JSON IDL
- **`lez_framework_cli::run()`** parses the IDL at runtime → provides the full typed CLI
- PDA seeds ensure deterministic account addresses per program ID
